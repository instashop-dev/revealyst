### Finding 1 (BLOCKER — privacy default) — shared-account flags bypass the visibility gate, leaking a real email in `private` mode

The W2 exit-gate item "privacy default verified as team-only pseudonymized"
is not true of the shipped code. The shared-account flags pipeline never
receives or checks `visibilityMode`:

- [`src/lib/dashboard-view.ts:50`](../../src/lib/dashboard-view.ts) calls
  `resolveSharedAccountSource().flags(scope, window)` — no `visibilityMode`
  argument, unlike the sibling `readDashboard`/`resolveSegmentSource` calls on
  the same line block, which both take it.
- [`src/lib/shared-account/index.ts:58`](../../src/lib/shared-account/index.ts)
  copies `subject.externalId` verbatim onto the flag, unpseudonymized.
- For Cursor, that `externalId` is literally `email:<real address>`
  ([`src/connectors/cursor/normalize.ts:41`](../../src/connectors/cursor/normalize.ts)).
- [`src/components/dashboard/shared-account-flags.tsx:56`](../../src/components/dashboard/shared-account-flags.tsx)
  renders `{flag.externalId}` directly to every signed-in org member — the
  dashboard page only calls `requireAppContext()`, no role gate.
- The gate's own audit predicate,
  [`assertTeamOnlyPseudonymized`](../../src/lib/visibility.ts) (line 46-66),
  only inspects `view.summary.scores` and `view.segments.segments[].members`
  — it never looks at `view.sharedAccounts`, even though `DashboardView`
  includes `sharedAccounts` as a top-level field. `tests/dashboard-privacy.test.ts`
  only asserts on `scores`/`segments`. **The test that's supposed to catch
  this is structurally blind to it** — it would pass even if every flag
  rendered a full name.

Reproduces on a realistic, non-malicious pattern (one engineer active across
≥16 distinct UTC hours in a day — no baseline-size guard on the
`round_the_clock` reason, unlike the volume heuristic).

**Disposition: fix now.** Two changes needed: (1) thread `visibilityMode`
into `SharedAccountSource.flags` and pseudonymize/redact `externalId` in
`private` mode (mirroring what `toPersonRef` does for people), and (2) extend
`assertTeamOnlyPseudonymized` to inspect `view.sharedAccounts` so the audit
predicate can no longer pass while this leaks.

**Fix applied.** `SharedAccountFlag.externalId` is now `string | null`;
`SharedAccountSource.flags` takes `visibilityMode` and nulls `externalId` in
`private` mode (`src/lib/shared-account/index.ts`); `dashboard-view.ts` now
forwards `visibilityMode` to the call it was missing it on;
`assertTeamOnlyPseudonymized` now also flags a non-null `sharedAccounts[].externalId`
as a leak (`src/lib/visibility.ts`); the panel falls back to a generic
"Shared account" label when redacted (`shared-account-flags.tsx`). Regression
coverage: `tests/shared-account.test.ts` (redaction in private vs. full mode)
and `tests/dashboard-privacy.test.ts` (extended to assert the fixture's
already-flagged shared account redacts in private, surfaces in full).

### Finding 2 (HIGH — consent) — any org member can publish a public share link for a teammate's score without their consent

[`POST /api/share`](../../src/app/api/share/route.ts) (line 22) only checks
that the target `personId` belongs to the caller's org
(`ctx.scope.people.get(body.personId)`) — not that the caller *is* that
person. Compare `src/app/api/reconcile/route.ts`, which is explicitly
`{ adminOnly: true }`; the share route has no such gate. A non-admin member A
who learns teammate B's `personId` (e.g. from `/api/people` or `/reconcile`,
both legitimately callable by A) can mint a real, public, unauthenticated
`/s/<token>` link showing B's actual score, labeled with any name A chooses —
with no notification to B, no consent from B, and no per-person listing of
"links about me" on B's own dashboard to discover it. ADR 0008 documents the
*intent* ("opt-in public link" for "the person whose card is shared") but the
API never enforces that the sharer is the subject. `tests/share-links.test.ts`
only covers cross-org isolation, never the same-org non-owner case.

**Disposition: fix now.** Add an ownership check to `POST /api/share`
(`body.personId` must resolve to the caller's own linked person, or require
`adminOnly` as an explicit interim gate) before this ships to any real team.

**Fix applied.** The route now loads the person and rejects with 403 unless
`person.authUserId === ctx.user.id` (`src/app/api/share/route.ts`), using the
existing `people.authUserId` column (no frozen-contract change — no
`org-scope.ts` method added). **Not covered by an automated route-level
test** — no route-handler test harness exists yet for either `/api/share` or
`/api/reconcile` (this is a pre-existing gap, not introduced here); building
that request-mocking scaffold is a separate follow-up, flagged but not done
in this pass.

### Finding 3 (BLOCKER — correctness/honesty) — stale `metric_records` rows from a vendor restatement are never deleted, permanently inflating live scores

`upsertRecords` ([`src/db/org-scope.ts:840`](../../src/db/org-scope.ts)) only
does `INSERT ... ON CONFLICT DO UPDATE` on the natural key (org, subject,
metric, day, dim) — it has no delete path for a key that was present in a
prior poll's batch but is absent from a later one. When a vendor restates a
trailing window and drops a `feature_used`/`model_tokens`/`model_requests` dim
for a (subject, day) — e.g. a reclassified or de-duplicated event — the
now-stale row from the earlier poll is never revisited or removed. It has no
expiry. `recomputeOrg`'s `distinct_dims` aggregation (the live
`adoption.tool_coverage` weight-0.5 and `fluency.breadth` weight-0.33 presets)
reads it directly, so the score stays permanently inflated with no
self-correction short of a manual out-of-band delete.

The guard for exactly this — [`deleteWindowForConnection`](../../src/db/org-scope.ts)
(line 899) — already exists, with a comment calling out this precise risk,
and is already wired into the agent-ingest (ChatGPT-export/CLI) path
(`src/lib/agent-ingest.ts:171`, delete-then-upsert) but was **never wired
into the poller** — `src/poller/run.ts:329` calls `upsertRecords` directly
with no preceding delete. No existing test exercises a second poll whose dim
set is a strict subset of the first (`connector-framework.test.ts`'s
"re-polls idempotently" test re-sends an identical batch).

**Disposition: fix now.** Wire `deleteWindowForConnection` (or an equivalent
delete-then-upsert) into `runConnectorPoll`/`runConnectorBackfill` before
gate sign-off — this silently and permanently inflates live scoring presets
with zero recovery mechanism, which is a direct instance of the "never
fabricate/inflate numbers" invariant.

**Fix applied.** `executeRun` (`src/poller/run.ts`) now calls
`scoped.metrics.deleteWindowForConnection(connection.id, window.start, window.end)`
immediately before `upsertRecords`, scoped to exactly the run's own
window/chunk — matching the already-shipped agent-ingest delete-then-upsert
pattern and the dispatcher's own comment ("vendors restate recent days ...
the upsert key makes re-polls overwrite," which this fix makes actually
true). Regression coverage: a new test in
`tests/connector-framework.test.ts` polls the same window twice with a
shrinking dim set (2 dims → 1) and asserts the dropped dim's row is gone
after the second poll, not just overwritten-in-place.
